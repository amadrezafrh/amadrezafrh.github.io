1. Overview

This project was developed as part of a national initiative to measure traffic flow through monitoring cameras installed at junctions in Tehran, Iran. The system uses YOLOv4 for vehicle detection, DeepSORT for multi-object tracking, and a semi-supervised approach combining Spectral Clustering with SVM for lane classification.

Note: This project uses YOLOv4 and was trained on limited data with augmentation. Many newer detection and tracking algorithms have since been developed with improved performance.

2. Pipeline

2.1. Detection and Tracking

A custom YOLOv4 model was trained on aerial vehicle imagery. DeepSORT algorithm handles multi-object tracking using a Kalman filter for state prediction and cosine distance for appearance matching.

The tracker maintains vehicle identities across frames using an 8-dimensional state space:

$$\mathbf{x} = [x, y, a, h, \dot{x}, \dot{y}, \dot{a}, \dot{h}]^T$$

where $(x, y)$ is the bounding box center, $a$ is the aspect ratio, $h$ is the height, and the remaining components are their respective velocities.

Association between detections and tracks uses a cost matrix combining appearance features (cosine distance) and motion information (Mahalanobis distance), solved via the Hungarian algorithm.

Real-time vehicle detection and tracking using YOLOv4 and DeepSORT

2.2. Lane Clustering

From tracking, three features are extracted for each detected vehicle and normalized using StandardScaler:

  • Horizontal position ($x$)
  • Vertical position ($y$)
  • Vehicle angle ($\theta$)

Spectral Clustering is applied to group vehicles by lane. Processing 1000 frames yielded approximately 60,000 vehicle detections with 180,000 total features. Various clustering algorithms were evaluated using silhouette score, with Spectral Clustering producing the best results.

Lane clustering using Spectral Clustering with 4 clusters

2.3. SVM Classification

To refine the clustering results and handle edge cases, an SVM with RBF kernel was trained on the labeled data from the clustering stage:

$$K(x_i, x_j) = \exp\left(-\gamma \|x_i - x_j\|^2\right)$$

The classifier takes normalized $(x, y)$ coordinates and predicts lane assignments, reducing errors in lane boundaries.

Refined lane classification after SVM training

2.4. Real-time Analysis

The final pipeline integrates all components for real-time video processing. The system classifies each detected vehicle into lanes and calculates per-lane statistics including vehicle count and average speed for traffic management applications.

Complete pipeline with lane classification and traffic analysis

3. Conclusion

This project demonstrates a complete pipeline for automated traffic flow measurement from aerial imagery. The combination of deep learning-based detection (YOLOv4), robust tracking (DeepSORT with Kalman filtering), and semi-supervised lane classification (Spectral Clustering refined with SVM) provides an automated solution for traffic monitoring at scale.

Behind the Build

Engineering Notes

Architecture, attribution for forked components, the algorithmic novelty in the clustering stage, and a 2026 take on what would be different if I rebuilt it today.

Context. B.Sc. thesis, IUST x Megamouj Co. Production deployment for national traffic monitoring in Tehran. Code last touched 2020-2021; treating this page as a retrospective rather than a live project.

Architecture

Four-stage pipeline. Stages 1 to 3 are one-off training; stage 4 is the real-time loop.

video
  -> YOLOv4 (custom-trained, TF SavedModel)
  -> DeepSORT tracker (Kalman + appearance metric)
  -> data.pkl  {frame_id: {track_id: {x, y}}}
       |
       v
  feature build  (x, y, heading)  over ~1000 frames
  -> Spectral Clustering            (lane discovery)
  -> SVM RBF on cluster labels      (smooth boundary)
  -> classifier.pkl + scaler_loc.pkl
       |
       v
  main.py real-time loop:
    YOLO -> DeepSORT -> SVM lane id
    -> per-lane count, mean speed, immovable cars
    -> OpenCV overlay

The novelty: lane discovery without labels

Labeling lane geometry on aerial footage is expensive. Instead, I let the data label itself by clustering tracked vehicle motion over a long window. Per car per frame I compute heading from the delta of two positions separated by dist frames, then build a standardised (x, y, angle) feature table.

for i in range(0, 900):
    for n in car_dic[i + 1]:
        x.append(car_dic[i + 1][n]['x'])
        y.append(car_dic[i + 1][n]['y'])
        if i > dist:
            for k in car_dic[i - dist]:
                if k == n:
                    dx = car_dic[i + 1][k]['x'] - car_dic[i - dist][k]['x']
                    dy = car_dic[i + 1][k]['y'] - car_dic[i - dist][k]['y']
                    if dx != 0 and dy != 0:
                        a = np.arctan2(dy, dx) * 180 / np.pi
                        if a < 0:
                            a += 360
                        angle.append(a)
                        x_prime.append(car_dic[i + 1][k]['x'])
                        y_prime.append(car_dic[i + 1][k]['y'])

df = pd.DataFrame({'angle': angle, 'x': x_prime, 'y': y_prime})
df = pd.DataFrame(StandardScaler().fit_transform(df),
                  columns=['angle', 'x', 'y'])

The spectral labels then become pseudo-ground-truth for an SVM with an RBF kernel, so inference is a single forward pass per detection instead of a re-cluster:

df = pd.read_csv('4_Spectral_Clusters_Cars.csv')
X = np.array(df[['x', 'y']])
Y = np.array(df['Spectral_labels'])

clf = SVC(kernel='rbf').fit(X, Y)

with open('classifier.pkl', 'wb') as f:
    pickle.dump(clf, f)

Real-time loop, key idea

In main.py the per-frame work after detection plus tracking is two scaler-transforms and a clf.predict, plus a small dictionary of running stats per lane.

X = scaler.transform(df_unscaled[['x', 'y']])
df_unscaled['class'] = clf.predict(X)

for lane_id, group in df_unscaled.groupby('class'):
    avg_speed   = int(group['speed'].mean())
    immovable   = (group['speed'] <= 6).sum()
    count       = len(group)
    overlay(frame, lane_id, count, avg_speed, immovable)

A 2026 take: this is mostly obsolete

The architecture made sense in 2020. Today the same pipeline can be a fraction of the code:

  • Detection. Replace the custom YOLOv4 weights with an open-vocabulary detector (Grounding DINO, YOLOv10-World) or prompt SAM 2 / SAM 3 with a natural-language target like "car from above". No bespoke training set needed.
  • Tracking. DeepSORT is still solid but BoT-SORT and ByteTrack are stricter on ID switches in dense traffic. Either drops in as a near-replacement for the deep_sort module.
  • Lane assignment. The whole Spectral-then-SVM stage exists because there was no foundation model that could read an intersection layout. Today a VLM (GPT-4o, Gemini, Qwen2-VL) prompted on a single frame can return a JSON list of lane polygons, removing the unsupervised stage entirely.
  • End-to-end. With a polygon list and a tracker, the rest of the pipeline collapses to point-in-polygon checks and running averages. No SVM, no scaler, no data.pkl training pass.

What I would still keep, and what I would add

  • Keep. The four-stage decomposition reads well as documentation; even with modern detectors I would still separate detection, tracking, lane structure, and aggregation so each is testable.
  • Add: proper evaluation. The original repo has silhouette and qualitative GIFs; for a redeployment I would add a small annotated test set with MOTA / IDF1 for tracking and lane-assignment accuracy.
  • Add: stream-friendly I/O. Replace OpenCV file-IO with an RTSP source and a small FastAPI service that publishes per-lane stats as JSON; the OpenCV overlay becomes a debug view.
  • Add: drift detection. Watch silhouette on the lane assignment over a rolling window; trigger re-clustering when the camera moves or the road layout changes, which the original pipeline could not detect.
